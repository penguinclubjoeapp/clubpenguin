# Room Editor — Design & Research

A web-based visual editor for creating and editing Club Penguin rooms in the Yukon client.

## Why we need this

The original rooms were built with two commercial tools:

- **Phaser Editor 2D** — visual scene layout (sprite placement, components, display lists)
- **PhysicsEditor by CodeAndWeb** — collision polygon drawing (walkable areas, trigger zones)

Several rooms are still unimplemented (triggers set to `null` in the codebase):

| Trigger | Reachable from | Notes |
|---|---|---|
| `boiler` | Dance, Cave | Boiler Room |
| `mine` | Cave, Shack | Mine |
| `pizza` | Plaza | Pizza Parlor |
| `stage1`, `stage2` | Plaza | Stage |
| `lighthouse` | Beach | Lighthouse |
| `ship` | Beach | Migrator |
| `lake` | Forest | Lake |
| `eco` | Shack | Eco area |
| `adopt` | Pet Shop | Puffle adoption (probably a widget) |
| `mix` | Dance | DJ mixing (probably a widget) |
| `dojohide` | Dojo Ext | Hidden dojo entrance (probably conditional) |

Adding these rooms requires artwork, physics bodies, scene layout, and wiring — a visual editor makes this feasible without commercial tools.

## How rooms work today

Each room consists of four pieces:

### 1. Server room definition — `yukon-server/data/rooms.json`

```json
{ "id": 808, "name": "mine", "member": 0, "maxUsers": 80, "game": 0, "spawn": 0 }
```

### 2. Client room metadata — `yukon/assets/media/crumbs/en/rooms.json`

```json
"808": { "key": "Mine", "path": "mine/Mine", "x": 760, "y": 680 }
```

### 3. Scene class — `yukon/src/scenes/rooms/mine/Mine.js`

Extends `RoomScene`. Defines:
- `this.roomTriggers` — maps physics body names to `triggerRoom(id, x, y)` calls
- `_preload()` — loads the asset pack
- `_create()` — places sprites, attaches Button/MoveTo/Animation components, defines sort order

Code between `/* START OF COMPILED CODE */` markers is generated by Phaser Editor 2D.
Custom logic lives in protected `/* START-USER-CODE */` regions that survive recompilation.

### 4. Asset files — `yukon/assets/media/rooms/mine/`

| File | Purpose |
|---|---|
| `mine-pack.json` | Asset pack manifest (tells Phaser what to load) |
| `mine.json` | Texture atlas frame definitions |
| `mine-0.png`, `mine-1.png` | Spritesheet images |
| `mine-physics.json` | Matter.js collision bodies (walkable area + trigger zones) |
| `mine-anims.json` | Optional animation definitions |

### 5. Scene source — `yukon/src/scenes/rooms/mine/Mine.scene`

The `.scene` JSON is the **source of truth** that Phaser Editor 2D reads and writes.
The `.js` file is compiled output. Example (`Forest.scene`):

```json
{
    "sceneType": "SCENE",
    "settings": {
        "superClassName": "RoomScene",
        "preloadMethodName": "_preload",
        "createMethodName": "_create",
        "sceneKey": "Forest",
        "preloadPackFiles": ["yukon/assets/media/rooms/forest/forest-pack.json"]
    },
    "displayList": [
        {
            "type": "Image",
            "label": "bg",
            "texture": { "key": "forest", "frame": "bg" },
            "x": -24, "y": -13,
            "originX": 0, "originY": 0
        }
    ],
    "lists": [
        { "label": "sort", "objectIds": ["..."] }
    ]
}
```

### Physics body format — `*-physics.json`

Generated by PhysicsEditor (CodeAndWeb). Consumed by `this.matter.add.fromPhysicsEditor()`.

Each room has:
- A `block` body — polygon defining the walkable area boundary (static collision walls)
- Named trigger bodies (e.g. `coffee`, `dock`, `plaza`) — sensor zones that fire `roomTriggers` callbacks when the penguin walks into them

```json
{
    "block": {
        "type": "fromPhysicsEditor",
        "label": "block",
        "isStatic": true,
        "density": 0.1, "restitution": 0, "friction": 0.1,
        "frictionAir": 0.01, "frictionStatic": 0.5,
        "collisionFilter": { "group": 0, "category": 1, "mask": 255 },
        "fixtures": [{
            "label": "",
            "isSensor": false,
            "vertices": [
                [{ "x": 1310, "y": 295 }, { "x": 831, "y": 262.5 }, ...]
            ]
        }]
    },
    "plaza": {
        "type": "fromPhysicsEditor",
        "label": "plaza",
        "isStatic": true,
        "fixtures": [{
            "isSensor": false,
            "vertices": [[{ "x": 41, "y": 330 }, { "x": 100, "y": 415 }, ...]]
        }]
    }
}
```

### Component system — `components.components`

Reusable behaviors attached to game objects in the scene editor:

| Component | Attaches to | Purpose |
|---|---|---|
| `Animation` | Sprite | Sprite animation (key, start/end frame, rate, repeat, autoPlay) |
| `Button` | Sprite | Hover frame swap + click callback |
| `SimpleButton` | Sprite | Hover callback only |
| `MoveTo` | Any | Defines where penguin walks when object is clicked |
| `Zone` | Rectangle | Invisible hover/click region |
| `ShowHint` | Any | Tooltip text on hover |
| `NineSlice` | Any | 9-slice texture rendering |
| `Draggable` | Any | Drag behavior |

## Recommended approach: Fork Phaser Editor 2D v3

**Repo:** [PhaserEditor2D/PhaserEditor2D-v3](https://github.com/PhaserEditor2D/PhaserEditor2D-v3)
**License:** MIT (frontend), proprietary (Go backend)
**Tech:** TypeScript frontend, Go backend (filesystem bridge)

### Why this is the right starting point

1. **We already use its exact output format** — every `.scene` and `.components` file in the codebase was created by this tool
2. **The entire frontend is MIT** — scene editor, component system, inspector, atlas browser
3. **The backend is trivially replaceable** — it's just file I/O over HTTP
4. **The User Components system already matches** — Button, MoveTo, Animation, Zone are already defined and the editor understands them
5. **Actively maintained** through 2025-2026 with Phaser 3.80+ support

### What the Go backend does (and how to replace it)

The proprietary backend is a filesystem bridge:
- Read/write project files
- Serve static assets (textures, spritesheets)
- License key validation (not needed — MIT frontend)

Replacement options:
- **Node.js/Express server** — ~200 lines, serves files and handles read/write
- **Browser File System Access API** — fully browser-native, no server needed

### What to build on top of the fork

| Feature | Effort | Description |
|---|---|---|
| **Replace Go backend** | 1-2 weeks | Express file server or browser File System Access API |
| **Physics polygon editor** | 2-4 weeks | Draw walkable areas and trigger zones over room backgrounds, output PhysicsEditor JSON |
| **Trigger zone editor** | 1 week | Extension of polygon editor for named sensor regions |
| **Strip unused features** | 1-2 weeks | Remove code editor, asset pack editor — make a focused room editor |

### Reference for the physics editor

[amuTBKT/Physics-Editor](https://github.com/amuTBKT/Physics-Editor) — lightweight open-source browser-based polygon editor with concave decomposition. Use as reference for the polygon drawing/vertex editing UX. The output format is simple vertex arrays — just needs to be wrapped in the PhysicsEditor JSON structure shown above.

## Alternatives considered

| Project | Why not |
|---|---|
| **GDevelop** (21k stars, MIT) | Different scene format, React+PixiJS+C++/WASM — total rewrite to output our `.scene` JSON |
| **Konva.js** (14k stars, MIT) | Canvas library, not an editor — would be building from scratch |
| **Tiled / LDtk** | Desktop only, tile-grid based — our rooms use free-form sprite placement |
| **Build from scratch** | Phaser Editor frontend already does 80% of what we need, MIT licensed |

## Architecture note: runtime scene loading

An alternative to code generation: write a **runtime `.scene` JSON loader** that builds Phaser scenes dynamically instead of compiling to `.js` files. This would mean:
- New rooms = just JSON + assets, no `.js` file needed
- The editor saves directly to the format the game consumes
- No compilation step
- Existing rooms could be migrated incrementally

This would be a separate effort from the editor itself but pairs well with it.
